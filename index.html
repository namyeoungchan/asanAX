THE ANALOGY

하나의 식당 안에 세 명의 주인공이 있습니다

웹사이트와 앱이 돌아가는 원리는 잘 굴러가는 식당의 ‘역할 분담’과 똑같아요. 셋이 손발을 맞춰야 음식 한 접시가 나옵니다.

손님 · Client

주문하는 사람

메뉴를 고르고, 음식이 나오면 받습니다. 주방 안 사정은 몰라도 돼요. ‘원하는 걸 말하고 결과를 받는’ 쪽.

주방·직원 · Server

요리하는 사람

주문을 받아 실제로 처리합니다. 창고에서 재료를 꺼내 조리하고 손님에게 내어줘요. ‘요청을 처리해 돌려주는’ 쪽.

창고·냉장고 · Database

재료를 보관하는 곳

면, 육수, 계란… 모든 재료가 정리돼 있어요. 손님과 직접 말하지 않고 오직 주방만 드나듭니다.

CLIENT · 손님

클라이언트는
‘내 눈에 보이는 모든 것’입니다

지금 보고 있는 이 화면, 인스타그램 앱, 카카오톡 채팅창. 손가락으로 만지고 눈으로 보는 부분 전부가 클라이언트예요. 식당의 ‘손님 자리’입니다.

  • 웹 브라우저(크롬·사파리)나 휴대폰 앱이 바로 클라이언트.
  • ‘좋아요’를 누르는 행동 = 주방에 보내는 주문(요청).
  • 요리는 못 만들어요. ‘요청하고 받아서 예쁘게 보여주는’ 역할.

주방에서 벌어지는 일 — 보이지 않는 처리 과정

SERVER · 주방

서버는
‘보이지 않는 곳에서 일하는 주방’

손님 눈엔 안 보이지만 모든 진짜 일은 여기서 일어납니다. 주문을 받고, 규칙대로 처리하고, 창고를 뒤져 결과를 만들어 손님에게 내보내죠. 24시간 불이 꺼지지 않는 컴퓨터입니다.

  • 항상 켜져 있는 강력한 컴퓨터. 내 노트북이 꺼져도 돌아가요.
  • ‘좋아요 1개 더해라’ 같은 규칙·계산을 처리하는 두뇌.
  • 손님과 창고 사이의 유일한 통로. 손님은 창고에 직접 못 가요.
DATABASE · 창고

데이터베이스는
‘잘 정리된 거대한 창고’

회원 정보, 올린 사진, 좋아요 숫자, 주고받은 메시지. 사라지면 안 되는 모든 것이 여기 보관됩니다. 그냥 쌓는 게 아니라 ‘게시글 88번 가져와’처럼 빠르게 찾을 수 있게 칸칸이 정리돼 있어요.

  • 전원이 꺼져도 데이터가 남아있는 영구 보관소.
  • 엑셀 표처럼 줄과 칸으로 정리돼 빠르게 찾습니다.
  • 손님을 직접 상대 안 해요. 오직 주방(서버)만 드나듭니다.

THE FULL JOURNEY

‘좋아요’ 한 번에 셋이 함께 움직입니다

아래 버튼을 눌러보세요. 주문서가 손님 → 주방 → 창고를 거쳐 다시 돌아오는 과정을 직접 따라가 볼 수 있어요.

“좋아요!” 주문을 보냄

② 주방 (서버)

주문 받고 처리 시작

③ 창고 (DB)

숫자 꺼내 +1 저장

버튼을 눌러 주문을 시작해 보세요.

CHEAT SHEET

한 줄로 정리하는 용어장

개발자와 대화할 때 이 표 하나면 충분합니다. 식당 비유와 함께 외워두세요.

클라이언트 Client / Front-end
손님 = 화면. 사용자가 직접 보고 만지는 부분. ‘프론트엔드’라고도 해요.
서버 Server / Back-end
주방 = 처리. 요청을 받아 실제 일을 하고 결과를 돌려주는 컴퓨터. ‘백엔드’라고도 해요.
데이터베이스 Database / DB
창고 = 보관. 데이터를 영구 저장하고 빠르게 찾아주는 곳. 서버만 접근합니다.
요청 Request
주문서. 클라이언트가 서버에게 보내는 ‘이거 해주세요’ 메시지.
응답 Response
나온 음식. 서버가 처리를 마치고 돌려주는 결과. 이걸 받아 화면이 바뀝니다.
API Interface
메뉴판 + 주문 규칙. 클라이언트와 서버가 약속한 ‘주문하는 방식’입니다.

이제 ‘클라이언트·서버·DB’가
한 식당으로 보이시나요?
